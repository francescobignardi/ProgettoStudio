Chiarito che oggi useremo solo (b) e (c) sul codice di Francesco, senza tirare giù librerie esterne, ma è la stessa infrastruttura che serve per Laravel.
- Francesco ha proposto una definizione intuitiva di namespace: "un po' i percorsi dei file, la rotta che mostra univocamente dove trovarli". Quasi giusta ma da correggere: il namespace è **una proprietà logica della classe** (dichiarata con `namespace ...;` in cima al file), non una proprietà del filesystem. L'allineamento con il path è **una convenzione PSR-4** che il progetto sceglie nel `composer.json` (`"App\\": "src/"`). Distinzione utile per Laravel dove le mappature non sono sempre banali (es. `App\Http\Controllers` → `app/Http/Controllers/`, non `src/`).
- Note di sintassi consegnate: `\` come separatore nel namespace, `/` nel path, doppio `\\` nel JSON di `composer.json` per via dell'escape.

**Esito (dopo)**

Risposta sintetica data (dettaglio a domani):

- **Docker su Mac = motore condiviso**: un'unica VM Linux ospita container e immagini di *tutto il computer*. Sopra al motore, i progetti sono isolati logicamente.
- **Sovrapposizioni inevitabili (visibili)**: Docker Desktop UI, pannello Docker dell'IDE, `docker ps`, RAM/CPU allocate, porte host. Container e immagini di ProgettoStudio appariranno accanto a quelli di lavoro nelle liste.
- **Isolamento garantito (non visibile)**: filesystem dei container, credenziali/env vars, ciclo di vita — un `docker compose down` in ProgettoStudio ferma solo i suoi.
- **Tre buone pratiche fondanti** (da applicare al setup di domani): (1) prefisso `progettostudio-` in tutti i `container_name`, (2) porte host non-collidenti (es. MySQL su `3316` invece di `3306`), (3) reti Docker separate (default di Compose per progetto).
- **Insidia dichiarata**: nella UI Docker Desktop non spegnere container senza verificare il nome — è il rischio più concreto. Il prefisso è la rete di sicurezza.
- **Analogia usata**: due macchine nello stesso parcheggio, entrambe presenti ma distinte dalla targa.
