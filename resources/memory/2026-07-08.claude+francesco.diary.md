## HH:MM — Fase 3, avvio Docker: il "perché" prima del "come"

**Intenzione (prima di agire)**
Come per Composer, prima del `docker-compose.yml` va il *perché*. Francesco parte da una base non nulla: (a) analogia immagine=classe / container=oggetto già chiara; (b) sa che Docker risolve problemi di compatibilità tra ambienti; (c) ipotizza (da verificare) che risolva anche il problema delle librerie condivise — cita l'esempio pre-Docker di An-Soft in cui la libreria Metis andava re-importata come `.phar` nelle dipendenze ogni volta che cambiava, e con Docker questo non succede più.

Piano di risposta:
1. Confermare/rifinire l'analogia immagine/container (giusta, con una precisazione).
2. Confermare il "problema di compatibilità" e nominarlo bene: **"works on my machine"**, dipendenze di sistema (versione PHP, estensioni, MySQL), non solo differenze di OS.
3. **Correggere/rifinire il pezzo sulle librerie**: qui c'è una confusione utile da sciogliere. Il problema del `.phar` di Metis re-importato non lo risolve **Docker** — lo risolve **Composer con un package manager** (dipendenza dichiarata + `composer update`). Docker risolve un problema diverso: quello dell'*ambiente in cui gira* la tua applicazione (PHP + estensioni + servizi come MySQL), non il modo in cui la tua applicazione consuma le librerie.
4. Introdurre i tre concetti base: **immagine / container / volume**. Compose come strumento per orchestrare più container insieme (nel nostro caso: php + mysql).

**Esito (dopo)**
- Francesco ha ricapitolato correttamente i quattro pilastri: (1) Docker = strato environment; (2) Composer = namespace + dipendenze esterne; (3) immagine = classe chiusa/readonly/template; (4) container = istanza runtime che vive e muore; (5) volume = persistenza; (6) Compose = orchestratore. Autovalutazione onesta: "livello base, ma se la base è giusta posso costruirci sopra". Base solida.
- Rifiniture minime consegnate: (a) il container non muore alla "chiusura della sessione" — muore quando lo fermi esplicitamente (`docker stop`), la CLI dell'host è un'altra cosa; (b) i volumi non "rendono persistenti modifiche a un container" — vivono in parallelo al container, non dentro; l'immagine resta immutabile.
- **Osservazione culturale di Francesco**: `composer.json` e `docker-compose.yml` sono nomi troppo simili — parte della confusione sui livelli deriva pure da quello. Punto valido: entrambi usano "compose" ma con significati diversi ("comporre le dipendenze" vs "comporre lo stack di container"). Buona da tenere presente.
