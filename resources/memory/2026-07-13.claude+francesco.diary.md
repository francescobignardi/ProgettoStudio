## 08:54 — Installazione Composer dentro il container

**Intenzione (prima di agire)**
Prossimo step: installare Composer dentro l'immagine PHP. L'immagine `php:8.3-cli` porta PHP ma non Composer — finora abbiamo usato Composer solo sul Mac (host) per `composer init` + `composer dump-autoload`, ma da qui in avanti serve **dentro** il container perché tutti i comandi Laravel (`composer create-project laravel/laravel backend`, `php artisan …`, install pacchetti) andranno lanciati nell'ambiente containerizzato.

Piano:
1. Spiegare *perché* Composer deve stare nel container, non sull'host (versione PHP coerente con quella di produzione, estensioni disponibili, riproducibilità).
2. Introdurre il pattern **multi-stage copy** da immagine ufficiale Composer: `COPY --from=composer:2 /usr/bin/composer /usr/bin/composer` nel Dockerfile. È il modo standard e più pulito per portare un binario da un'immagine all'altra senza rifare l'installazione manualmente.
3. Rebuild + verifica con `composer --version` dentro il container.

Concetto nuovo da fissare: `COPY --from=<altra-immagine>` è una feature dei Dockerfile che permette di attingere file da immagini esterne come se fossero build stage. Molto usato per portare binari pre-compilati (Composer, `mc`/minio client, `dockerize`, ecc.) senza appesantire il Dockerfile con install da sorgenti.

**Esito (dopo)**
- Prima passata mi sono espresso male ("aggiungiamo Composer *all'immagine*") e Francesco l'ha letta come "aggiungiamo `composer.json` all'immagine". Confusione tool-vs-file da chiarire subito. Chiarimento in due giri: (1) primo, verboso, con tabella tool/codice; (2) secondo, richiesto esplicitamente da Francesco in versione compatta e priva di termini complessi. La seconda versione è quella che ha fatto scattare la comprensione.
- Concetto chiave fissato: `composer` (il tool) sta nell'immagine tramite `COPY --from=composer:2`; `composer.json` (il file) sta nel container tramite bind mount dichiarato nel compose (la cartella progetto è "vista in tempo reale" da `/app`). Analogia efficace: "il cuoco vs la ricetta". Divisione più generale: **tool** nell'immagine (via Dockerfile), **codice del progetto** nel container (via bind mount).
- Riga `COPY --from=composer:2 /usr/bin/composer /usr/bin/composer` aggiunta al Dockerfile. Build in 4.1s: layer `RUN` estensioni **cachato** (0.0s), solo il nuovo layer `COPY` ricompilato. Prima volta in cui la cache dei layer si vede in vivo, buon momento didattico.
- Verifica: `docker exec progettostudio-php composer --version` → `Composer version 2.10.2 ... PHP version 8.3.31 (/usr/local/bin/php)`. Nota importante: Composer sta usando *proprio il PHP del container*, non quello del Mac — è esattamente il motivo per cui l'abbiamo installato dentro. Da qui in avanti tutti i `composer install/require` gireranno nell'ambiente containerizzato.
- Lezione mia da tenere: quando parlo di "Composer" senza specificare, il junior può confondere tool vs config. La coppia binario/file è ovvia solo dopo. Preferire nomi disambiguati ("il programma composer" / "il file composer.json") almeno finché la distinzione non è solida.

## 09:26 — Installazione dello scheletro Laravel in `backend/`

**Intenzione (prima di agire)**
Ora che il container ha PHP 8.3 + estensioni Laravel-ready + Composer, arriva il pezzo grosso: scaricare lo scheletro Laravel. Il comando è `composer create-project laravel/laravel backend`, da lanciare **dentro il container** con `docker exec`.

Cosa fa `composer create-project`:
- È diverso da `composer install`. Non installa dipendenze di un progetto esistente — **crea un nuovo progetto** a partire da un template pubblicato su Packagist (in questo caso `laravel/laravel`, che è lo scheletro ufficiale).
- Scarica il template nella cartella indicata (`backend`), poi lancia automaticamente `composer install` per tirare giù tutte le dipendenze.
- Alla fine avremo `backend/` popolata con la struttura standard Laravel (app/, config/, routes/, database/, ecc.) + un `backend/vendor/` con ~40 pacchetti + un `backend/.env` di default.

Concetto da fissare: differenza tra `composer create-project` (crea nuovo progetto da template) e `composer install` (installa dipendenze di un progetto esistente leggendo il `composer.json` che c'è già). Sono i due gesti fondamentali di Composer, spesso confusi.

Piccola scelta da fare prima di lanciare: la versione di Laravel. Se non specifichiamo, prende l'ultima stabile. Per un progetto di studio nuovo va benissimo — al 2026 dovrebbe essere Laravel 12.x.

**Esito (dopo)**

Cartella `backend/` popolata con struttura Laravel standard (app/, bootstrap/, config/, database/, public/, resources/, routes/, storage/, tests/, vendor/) + file `artisan`, `composer.json`, `composer.lock`, `README.md`, `package.json`, `vite.config.js`, `phpunit.xml`. Nota: Laravel 11+ include di default il setup Vite per asset frontend, non ci servirà finché il frontend resta Next.js separato.
- Automatismi post-install di Laravel: (a) `key:generate` ha popolato `APP_KEY` nel `.env`; (b) creato `database/database.sqlite` vuoto; (c) `artisan migrate --graceful` ha eseguito le tre migrazioni base (users, cache, jobs) contro **SQLite** — è il default dello scheletro Laravel dal 10.x in poi, per abbassare la barriera d'ingresso.

- Riflessione di Francesco sulla DX moderna vs framework custom aziendali: `composer create-project laravel/laravel` genera setup completo (chiavi, DB migrato, config default) in 30s — nel framework custom di lavoro (An-Soft) ogni progetto si compone a mano pezzo per pezzo copiando da altri già fatti, con zero documentazione.
