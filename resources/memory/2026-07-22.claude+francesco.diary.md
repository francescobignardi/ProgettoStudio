**Contesto che F. porta già** (ha studiato ieri per conto suo)
- Ha capito cos'è Eloquent: un **ORM** (Object-Relational Mapping), mappa tabelle → oggetti, permette operazioni DB via codice invece di SQL a mano. Lo vede comodo e "sicuro" (errori macroscopici più difficili).
- Ricordo doloroso: le **Laravel migrations** viste brevemente al lavoro — togliere una PK o una colonna era "un bagno di sangue", mentre su SequelAce è banale. Con MySQL diretto si è sempre trovato bene.
- Atteggiamento aperto: si aspetta che Eloquent abbia pregi che imparerà ad apprezzare.

**Lezioni date durante la ripresa**
- Il "sicuro" di Eloquent è concreto: query parametrizzate (PDO) → niente SQL injection / escaping a mano. Più: il codice parla di dominio, non di tabelle.
- Migrations vs SequelAce = due mestieri diversi. SequelAce *modifica lo stato*; le migration *descrivono una trasformazione ripetibile e versionata*. Il valore delle migration è per team + deploy + storia, non per il singolo su un DB. Il dolore del `down()` chirurgico nasce dall'applicare rigore-da-produzione a un contesto dev dove basta `migrate:fresh`.

- **Collisione EO ↔ Laravel discussa esplicitamente** (cfr. [[feedback_yegor_reshape]]): Eloquent = pattern **Active Record**, oggetto-guscio permissivo (`new Product()` vuoto lecito), riempito via array. EO = costruttore esplicito che protegge gli invarianti. Nessun vincitore: due filosofie, prezzi diversi. F. ha espresso preferenza per architetture pulite dove il framework fa il dietro-le-quinte (vs framework custom del titolare, verboso e opaco). Criterio dato: il framework deve nascondere ciò che è **noioso e prevedibile**, non ciò che è una **decisione** (il `$fillable` è verbosità *buona*).
- Falso positivo IDE #2: "`create` not found in Product" → `create()` è di `Model` via magic method `__call`; l'analisi statica non vede la magia dei framework. **NON** fare "Add method" (sovrascriverebbe Eloquent). Cura nota: Laravel IDE Helper (non installato ora).
- `make:seeder ProductSeeder` (rinominato da `ProductSeed` via refactor→rename: verificata doppia coerenza classe+file per PSR-4). 4 prodotti via `Product::create([...])` — uno per riga (un array = una riga). Registrato in `DatabaseSeeder` con `$this->call(ProductSeeder::class)`. `php artisan db:seed` OK, DB popolato (id auto, timestamps auto, price decimale esatto).
- **Idempotenza — lezione chiave**: i seeder **non hanno memoria** (nessuna tabella di tracciamento, a differenza di `migrations`). Rilanciare `db:seed` **riesegue tutto** e accumula. F. l'ha visto dal vivo: 2° `db:seed` → prodotti raddoppiati a 8 + `UniqueConstraintViolationException` sul `User` di default (`test@example.com`, colonna unique). `migrate:fresh` **non** è un rollback: droppa TUTTE le tabelle ignorando i `down()`, rilancia tutte le migration da zero, `--seed` ripopola → 4 prodotti puliti id 1-4. È la risposta definitiva al "bagno di sangue": in dev si rade al suolo, non si fa rollback chirurgico. **Mai `fresh` in produzione** (cancella tutti i dati).

- **Lezione difensivo vs fail-fast**: F. aveva aggiunto `@isset($products)` per abitudine (best practice introiettata). Discusso: `$products` è garantito dal contratto controller→view → difendersi da una garanzia *nasconde* i bug invece di esporli. Criterio trasversale: spendi verbosità dove c'è vera incertezza, non dove c'è garanzia (stesso criterio del `$fillable`, rovesciato).
- Intoppo Blade risolto: `@empty($products)` con parentesi dentro `@forelse` → ParseError "unexpected end of file" (le parentesi aprono un *altro* costrutto che vuole `@endempty`). Fix: `@empty` **nudo**.
